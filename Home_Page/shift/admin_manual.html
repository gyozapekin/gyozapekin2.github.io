マスターパスワード機能

設定タブで「管理者マスターパスワード」を設定すると、その値で全スタッフアカウントにログインできます。スタッフがパスワードを忘れた時の代理ログイン用です。

マスターパスワードは管理者だけが知っている強力な値にしてください。他人に知られると全アカウントに不正アクセス可能になります。

3-4. 日付カードのその他

  • 右上の日付の左に「定休にする／定休に戻す／営業にする」ボタン
  • 定休日は灰色背景、臨時休業日は薄赤背景＋「臨」マーク
  • セル下部の色付きドット：その日の希望提出状況（緑=多い／黄=普通／赤=少ない）
編集モードを抜けると自動保存されます。「保存」ボタンはありません。Firebase に即時反映されます。

4. 希望カレンダー

スタッフ希望と北京の希望を月単位で見渡す画面です。誰がどの日に何を希望しているか、ひと目で把握できます。

上部ツールバー
◀ 2026年 7月 ▶ 時間帯： 昼 夜 両方
フィルター： ○ △ ×
凡例： ○出勤可 ×休み希望 △調整可 ☀昼 🌙夜

使い方のコツ

  • 昼／夜タブで時間帯を絞ると、その時間帯だけの候補が見やすい
  • 「△調整可」だけを表示すると、空きが出た時に頼める人がわかる
  • 北京の希望（店から）は紫色の枠で表示
  • フィルターの設定はブラウザに保存されます

6. スタッフ管理

6-1. スタッフ一覧

登録されているスタッフが一覧表示されます。種別ごとに色分け。

種別意味
固定シフトが固定されている社員・常勤

6-2. 新規追加

  1. 「+ スタッフを追加」ボタン
  2. 名前（表示名）／ID（英字、ログイン用）／初期パスワード／種別を入力
  3. 「追加」で確定

6-3. 編集

各スタッフ行の「編集」ボタンから名前・種別・パスワードを変更可能。

6-4. 削除（アーカイブ）

削除ボタンは完全削除ではなく退職フラグです。過去のシフト・希望データは保存されます。集計や履歴で必要なので、完全削除しない設計です。

6-5. 退職者表示切替

上部のラジオで「現役のみ」「全員（退職者含む）」を切り替えられます。退職者を再雇用する場合は「全員」表示にして編集→アーカイブ解除。

7. 設定（締切設定含む）

7-1. 定休日の曜日設定

毎週決まった休業日（デフォルト：火・水）。チェックを切り替えるとカレンダーに反映されます。

7-2. 特別休日・臨時休業

特定日付の休業（祝日振替、棚卸し、店長旅行等）。「+ 日付追加」で登録。カレンダーで該当日に「臨」マークが表示されます。

7-3. 定休日の営業例外日

本来定休日だが営業する日（特別営業）。逆方向の例外設定。

7-4. 管理者マスターパスワード

全スタッフアカウントに使える特別パスワード（前述）。

7-5. シフト希望締切設定（重要・新機能）

スタッフが何月分のシフト希望を出せるかをコントロールします。デフォルトは前月の20日。月別・スタッフ別に上書き可能。

階層構造

締切は次の順に判定されます（上ほど優先）：

  1. 個人別の上書き（特定スタッフだけ別の締切）
  2. 月別の上書き（その月全体の締切を変更）
  3. デフォルト（前月の20日 23:59）

UIサンプル

月別の締切一覧
2026年 7月 デフォルト: 6/20 保存
2026年 8月 上書き中: 7/25 保存↺ リセット
👤 個別

個別調整（月別→特定スタッフ）

「👤 個別」ボタンを押すと、その月のスタッフ別締切パネルが展開します。

個別調整パネル（2026年7月）
スタッフ別の締切調整 ✕ 閉じる
ふさちゃん 標準: 6/20 保存
みきちゃん 上書き: 6/25 保存↺ リセット
「変更なし」は保存しても上書きが増えません。うっかり開いて「保存」を押しても何も書き込まれない設計です。値を変えた時だけ上書きとして記録されます。

使用例

  • 連休・繁忙月に「全員の締切を5日延ばす」→ 月別の日付ピッカーで変更→保存
  • 特定スタッフだけ「旅行から戻った日まで延長」→ 個別パネルでその人だけ変更
  • 普段通りに戻したい → 「↺ リセット」

10. 公開フロー（運用ガイド）

ベストプラクティス

  1. テンプレートで下書き作成：テンプレートタブ→「今月に反映」
  2. シフト管理で調整：編集モードに入る→候補から選んで確定／外す
  3. 統計で偏りチェック：連勤・占有率を確認
  4. 公開プレビュー：スタッフ視点で内容確認
  5. 公開する：確認ダイアログでOK
  6. スタッフへ通知：LINE等で「2026年7月のシフトが公開されました」

後で修正が必要になったら

編集モード→修正→もう一度「公開する」で再公開（上書き）。スタッフ画面にも自動反映されます。

未公開に戻すケース

  • 大幅な修正が必要で、いったん全部見直したい時
  • 誤って公開した時

「未公開に戻す」を押すと、スタッフ画面では「まだ公開されていません」表示に戻りますが、下書きは残ったままです。

11. リアルタイム同期と編集ロック

11-1. リアルタイム同期

同じ画面を複数の端末で開いていても、Firebase 経由で自動同期されます。誰かがシフトを変更すると、他の端末にも即座に反映され、画面下にトースト通知が出ます。

11-2. 編集ロック（排他制御）

他の管理者が編集中
⚠ 管理者B が編集中です
最終操作: 12秒前　[引き継ぐ] [閉じる]
  • 1人が編集モードに入ると、他の管理者は閲覧モードになる
  • 30秒以上反応がない場合、「引き継ぐ」ボタンで強制的に編集権を奪取できる
  • 編集中のタブを閉じる（ブラウザ閉じ）と自動的にロック解放

12. セキュリティ・パスワード

12-1. パスワードの保存

パスワードはSHA-256 + 個別ソルトでハッシュ化されてFirebaseに保存されます。平文パスワードは保存されません。

12-2. Firebase認証

すべての読み書きは Anonymous Auth トークンが必要です。Firebase Rules で auth != null を要求しているため、未認証アクセスは拒否されます。

12-3. 環境分離

環境URLFirebase Namespace
本番gyozapekin.com/shift//pekin/db/
Devgyozapekin.com/gyozapekin2.github.io/shift//pekin/db_dev/
本番とDevはFirebaseのネームスペースで完全に分離されています。Devでのテストが本番に影響することはありません。

13. トラブルシュート

再同期してもデータが古い
一度ログアウト→ログイン。それでも改善しない場合、ブラウザのキャッシュを Ctrl+Shift+R で強制リロード。
公開ボタンを押しても反映されない
F12 でブラウザの開発者ツールを開き、Console タブで Firebase エラーが出ていないか確認。「permission denied」が出る場合は Firebase Rules またはAuth設定の問題。
スタッフが「受付終了」で希望を出せない
設定→「シフト希望締切設定」で該当月の締切を延長。または個別パネルでそのスタッフだけ延長。
編集モードに入れない
他の管理者がロック中。30秒以上経過していれば「引き継ぐ」ボタンが出ます。それでも入れない場合、相手の端末でブラウザを閉じてもらってください。
スタッフのパスワードを忘れたと言われた
スタッフ管理タブで該当スタッフを「編集」→パスワードを新規入力。または、マスターパスワード機能を使えば管理者側でリセット不要に。
テンプレート反映を間違えて押した
編集モードに入って「↶ 戻る」で取消。または再度テンプレートを修正→「今月に反映」で正しい状態を再生成。
公開済みのシフトを修正したい
編集モード→修正→もう一度「公開する」で上書き再公開。スタッフ画面に即時反映されます。
Dev環境で操作した内容が本番に出てしまった
原理的に起こりえません。Firebaseネームスペースが分離されています。もし本当に起きていたら、URL を再確認してください（gyozapekin2.github.io が含まれているか）。
